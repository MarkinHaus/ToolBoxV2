LiveSync Internals

Architecture, protocol, and module reference for developers extending or debugging the LiveSync system.

Component Map

LiveSync consists of three independent runtime components communicating over WebSocket (notifications) and S3 API (data).

SyncService (server.py)
WS server, auth + credential provisioning, broadcast engine, server-side index, watchdog on server vault. Runs as a standalone subprocess.
MinIO
Persistent encrypted file store. Per-client scoped credentials via CredentialBroker. Clients upload/download directly — server is NOT in the data path.
ToolBoxV2 (__init__.py)
Supervisor: starts/stops SyncService subprocess, healthcheck, share CRUD. Provides @export API for TB integration.
SyncClient (client.py)
Runs on each device. Watchdog → debounce → encrypt → MinIO upload → WS notify. Receives WS notifications → MinIO download → decrypt → atomic write. Auto-reconnect with exponential backoff. Offline buffer for changes during disconnection.
Shared Libraries
protocol.py, config.py, crypto.py, index.py, minio_helper.py, conflict.py — used by both server and client.
Key Invariant

WebSocket NEVER carries file content. Only metadata + MinIO object keys. All file data flows directly between clients and MinIO.

Data Flow — Syncing a File

Client A: Watchdog detects file change
↓
DebounceBatch: 2s collect, dedup by path
↓
Checksum check against local index (skip if unchanged)
↓
zlib.compress() → AES-256-GCM encrypt → MinIO PUT
↓
WS → Server: {"type":"file_changed", "path", "checksum", "minio_key"}
↓
Server: update index, check for conflict, broadcast to Client B
↓
WS → Client B: {"type":"file_changed", ..., "source_client":"A"}
↓
Client B: MinIO GET → AES decrypt → zlib decompress → atomic write
↓
Client B: verify SHA-256 checksum → update local index

Process Model

// Independence: processes survive each other's crashes

ToolBoxV2 starts → SyncService as subprocess (own port)
SyncService crash → ToolBox detects → auto-restart
ToolBox crash    → SyncService keeps running (independent process)
Client crash     → auto-reconnect with exponential backoff (1s → 60s max)

MinIO Credential Flow

Clients never have admin credentials. The server mints scoped service accounts via CredentialBroker at auth time.

Client sends WS: auth {client_id, share_id}
↓
Server calls CredentialBroker.vend_share_credentials(share_id)
↓
MinIO Admin API: create service account with scoped policy
↓ Policy: s3:*Object + s3:ListBucket on livesync/{share_id}/*
Server sends WS: auth_success {minio_credentials, checksums}
↓
Client initializes Minio(creds) → direct S3 uploads/downloads
Security Design

The Share Token contains the AES encryption key + endpoints but never MinIO credentials. Credentials are provisioned per-session via WebSocket after authentication. If CredentialBroker fails, the system falls back to admin credentials with a logged warning.

Watchdog → Asyncio Bridge

Watchdog runs callbacks on its own thread. Calling asyncio.create_task() from there crashes the event loop. The fix:

class AsyncWatchdogHandler(FileSystemEventHandler):
    def __init__(self, loop, queue, vault_path):
        self.loop = loop      # asyncio event loop
        self.queue = queue    # asyncio.Queue

    def on_modified(self, event):
        # Thread-safe! Never asyncio.create_task() here
        self.loop.call_soon_threadsafe(
            self.queue.put_nowait, ("modified", rel_path)
        )python

The asyncio side drains the queue on each loop iteration, feeds events into DebounceBatch, and processes after the 2-second window.

Bug Prevention

The client also suppresses watchdog events for files it's currently writing (_writing_paths set). Without this, downloading a file would trigger a re-upload loop.

Conflict Resolution

Detection

Server compares incoming checksum against its index. If both are non-empty and differ → conflict.

def detect_conflict(local_cs, remote_cs) -> bool:
    if not local_cs or not remote_cs:
        return False   # new file, no conflict
    return local_cs != remote_cspython

Resolution Strategy

File TypeStrategyImplementation
.mdMerge markersBoth versions embedded with <<<<<<< LOCAL / ======= / >>>>>>> REMOTE
BinaryLatest-winsNewer mtime wins. Loser saved as .conflict.{checksum}.ext
AllBackup-first.backup created before any overwrite
DeletesTrashMoved to .sync-trash/, never permanently deleted
Safety Invariant

Every conflict is logged (sync_log table), broadcast to all clients (WS conflict message), and printed to console. No silent data loss.

Full State Sync (100k Files)

Sending 100k checksums as JSON over WebSocket would be ~4MB of JSON. Instead:

New client sends: request_full
↓
Server: index.export_gzipped() → SQLite dump → gzip
↓
Server uploads to MinIO: share/.meta/index.db.gz
↓
WS: full_state_ready {minio_key, file_count}
↓
Client downloads DB → import_gzipped() → diff → batch download (max 5 concurrent)

Test Philosophy

All tests use unittest (no pytest). MinIO and WebSocket are mocked for unit tests. Server/client tests verify logic without starting real network services. Integration testing against a real MinIO instance is a future step.
