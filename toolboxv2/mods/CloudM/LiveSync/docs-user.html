LiveSync Benutzer­handbuch

Datei auf dem Desktop ändern — in Sekunden auf dem Handy. LiveSync hält deine Obsidian-Vaults über beliebig viele Geräte synchron.

Echtzeit-Sync Client-seitig verschlüsselt Self-hosted Kein Datenverlust

Voraussetzungen

LiveSync läuft komplett self-hosted auf deiner ToolBoxV2-Infrastruktur. Du brauchst:

KomponenteDetails
ServerToolBoxV2 mit MinIO (läuft bereits auf dem RYZEN-Server)
DesktopPython 3.10+, beliebiges OS
HandyTermux (Android) mit Python
NetzwerkPort 8765 (WebSocket) + Port 9000 (MinIO) erreichbar

Installation

Server (einmalig)

# Abhängigkeiten installieren
pip install websockets watchdog minio cryptography aiosqlite pydantic

# Selftest — prüft ob alles vorhanden ist
python -c "from toolboxv2.mods.CloudM.LiveSync import run_selftest; print(run_selftest())"
{'websockets': True, 'watchdog': True, 'minio': True, ...}

Client (jedes Gerät)

# Gleiche Abhängigkeiten
pip install websockets watchdog minio cryptography aiosqlite pydantic
Termux (Android)

In Termux zuerst pkg install python rust ausführen, dann die pip-Pakete installieren. cryptography braucht den Rust-Compiler.

Share erstellen

Ein Share verbindet einen lokalen Ordner mit einem verschlüsselten MinIO-Store. Du erstellst ihn auf dem Server, dann treten andere Geräte mit dem Token bei.

Server starten

python -m toolboxv2.mods.CloudM.LiveSync.server \
    --vault /srv/obsidian/vaults/main \
    --share-id obsidian-vault \
    --port 8765

Share + Token generieren

python -c "
from toolboxv2.mods.CloudM.LiveSync import create_share
result = create_share('/srv/obsidian/vaults/main', ws_port=8765)
print('Token:', result['token'])
"
Token: eyJ2IjogMSwgInNoYXJlX2lkIjogImFiYzEyMyIsIC4uLn0=

Token verteilen

Kopiere den Token-String an dein anderes Gerät (per Message, QR-Code, Clipboard — der Token enthält keine MinIO-Passwörter, nur Endpoint + Verschlüsselungskey).

Sicherheit

Der Token enthält den AES-256 Verschlüsselungsschlüssel. Wer den Token hat, kann die Dateien lesen. Teile ihn nur über sichere Kanäle.

Share beitreten

Auf jedem weiteren Gerät (Laptop, Termux, …):

python -m toolboxv2.mods.CloudM.LiveSync.client \
    --token "eyJ2IjogMSwgInNoYXJlX2lkIjogImFiYzEyMyIsIC4uLn0=" \
    --vault ~/obsidian-vault

Der Client verbindet sich per WebSocket, erhält MinIO-Zugangsdaten, und beginnt sofort mit dem Sync. Beim ersten Start werden alle Dateien heruntergeladen.

Sync stoppen

# Einzelnen Share stoppen
python -c "from toolboxv2.mods.CloudM.LiveSync import stop_share; print(stop_share('abc123'))"

# Gesamten Server stoppen
python -c "from toolboxv2.mods.CloudM.LiveSync import stop_sync; print(stop_sync())"

Der Client stoppt mit Ctrl+C. Ungesendete Änderungen werden beim nächsten Start nachgeholt.

Wie Sync funktioniert

LiveSync nutzt ein Notification-Modell: Dateien gehen direkt über MinIO, WebSocket transportiert nur Metadaten.

Datei ändern
→
Watchdog
→
2s Debounce
→
zlib + AES → MinIO
→
WS Notify
→
MinIO → AES + zlib
→
Datei da ✓
Performance

Änderungen werden 2 Sekunden gesammelt (Debouncing), dann als Batch verarbeitet. Maximal 5 gleichzeitige Transfers. Bei einem Vault mit 100.000 Dateien wird der initiale Sync als komprimierte SQLite-DB übertragen.

Konflikte lösen

Wenn zwei Geräte gleichzeitig dieselbe Datei ändern, erkennt LiveSync den Konflikt und reagiert je nach Dateityp:

DateitypStrategieWas passiert
.md Merge-Marker Beide Versionen werden in die Datei geschrieben, getrennt durch Git-Style-Marker. Du entscheidest manuell.
.png .jpg .pdf Latest-Wins Die neuere Version gewinnt. Die ältere wird als .conflict-Backup gespeichert.

Merge-Marker Beispiel

<<<<<<< LOCAL (desktop-markin @ 18:30:01)
Hier steht die Version von deinem Desktop.
=======
Hier steht die Version vom Handy.
>>>>>>> REMOTE (handy @ 18:30:00)

Öffne die Datei, wähle die richtige Version, lösche die Marker, speichere — LiveSync synct die bereinigte Datei automatisch.

Wichtig

Konflikte werden niemals stillschweigend gelöst. Du siehst immer eine Log-Meldung und einen Eintrag im Sync-Log.

Gelöschte Dateien

Wenn eine Datei auf einem Gerät gelöscht wird, wird sie auf den anderen Geräten nicht sofort gelöscht — stattdessen wandert sie in den Ordner .sync-trash/ im Vault.

vault/
├── notes.md
├── .sync-trash/
│   └── 1713379200_old_notes.md    ← hierhin verschoben
└── ...

Du kannst jederzeit Dateien aus .sync-trash/ wiederherstellen.

Offline-Nutzung

Du kannst normal weiterarbeiten, auch wenn das Netzwerk weg ist:

Offline arbeiten

Dateien normal bearbeiten. LiveSync merkt sich alle Änderungen lokal.

Wieder verbinden

Client reconnected automatisch (exponential backoff: 1s, 2s, 4s, …, max 60s).

Catchup

Checksums werden verglichen. Fehlende Dateien werden heruntergeladen, lokale Änderungen hochgeladen.

Fehlerbehebung

[LiveSync] ERROR MinIO unreachable

MinIO ist nicht erreichbar. Prüfe: läuft systemctl status minio? Ist Port 9000 offen? Sind die ENV-Variablen MINIO_ENDPOINT, MINIO_ROOT_USER, MINIO_ROOT_PASSWORD korrekt gesetzt?

[LiveSync] ERROR Checksum mismatch after 3 retries

Die heruntergeladene Datei stimmt nicht mit der erwarteten Prüfsumme überein. Meist ein Netzwerkproblem. Lösche die lokale .sync-tmp-Datei und starte den Client neu.

[LiveSync] WARN Conflict detected

Kein Fehler — zwei Geräte haben gleichzeitig dieselbe Datei geändert. Siehe Konflikte lösen.

Client reconnected nicht

Prüfe ob der Server läuft und Port 8765 erreichbar ist. Der Client versucht automatisch mit steigenden Wartezeiten (bis 60s) zu reconnecten.

Niemals manuell die SQLite-DBs löschen

Die Dateien .livesync_client.db und .livesync_server.db enthalten den Sync-Index. Löschst du sie, muss ein vollständiger Resync durchgeführt werden.
